## Выводы

### Когда эффективнее Insertion Sort

| Ситуация | Причина |
|---|---|
| Данные **уже отсортированы** | Работает за O(n) — внутренний цикл не выполняется вообще |
| Данные **почти отсортированы** | O(n · k), где k — число инверсий; при малом k очень быстро |
| **Малые массивы** (n < ~500) | Меньше накладных расходов, простые обращения к памяти |
| **Поточная вставка** | Можно добавлять элементы по одному без перестройки всей структуры |

### Когда эффективнее Heap Sort

| Ситуация | Причина |
|---|---|
| **Случайные данные**, n > 500 | O(n log n) против O(n²) — разрыв растёт с ростом n |
| **Обратно упорядоченные** данные | Insertion деградирует до худшего случая; Heap нечувствителен к порядку |
| **Неизвестный порядок** входа | Гарантирует O(n log n) при любых данных |
| Нужен **детерминированный** результат | Нет деградации, как у QuickSort |

### Итоговая таблица

```
Входные данные        n < ~500      n > ~500
───────────────────────────────────────────────
Случайные             Insertion     Heap  ← в 200× разница при n=50000
Возрастание           Insertion     Insertion  ← O(n), Heap всегда O(n log n)
Убывание              Heap          Heap  ← Insertion даже при n=5000 в 59× медленнее
Неизвестный порядок   Insertion     Heap
```

> **Практика:** гибридный подход — Insertion Sort для подмассивов ≤ 16–32 элементов,
> Heap Sort / MergeSort для больших. Именно так работает `std::sort` (Introsort).
