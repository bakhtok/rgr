**Ахмедов Бахтовар Боирович, вариант 2**  
Алгоритмы: сортировка вставками · пирамидальная сортировка  
Реализация: обобщённый интерфейс `void*, qsort`-совместимый, динамическая библиотека `libsort.dylib`

---

## 1. Описание алгоритмов

### Сортировка вставками (Insertion Sort)
На каждом шаге `i` берётся элемент `arr[i]` и вставляется на нужное место в уже отсортированной части `arr[0..i-1]` путём последовательного сдвига элементов.

```
ключ = arr[i]
j = i - 1
пока j >= 0 и arr[j] > ключ:
    arr[j+1] = arr[j]
    j--
arr[j+1] = ключ
```

### Пирамидальная сортировка (Heap Sort)
1. Построение max-кучи из массива за O(n) (итеративный `siftDown`).
2. Повторное извлечение максимума: обмен корня с последним, `siftDown` по уменьшенной куче.

---

## 2. Теоретическая сложность

| Алгоритм | Лучший случай | Средний случай | Худший случай | Память |
|---|---|---|---|---|
| Insertion Sort | **O(n)** — уже отсортирован | O(n²) | O(n²) — обратный порядок | O(1) |
| Heap Sort | O(n log n) | O(n log n) | O(n log n) | O(1) |

Оба алгоритма работают **на месте** — дополнительная память O(1) (не считая буфера размером с один элемент).

## 4. Анализ

### 4.1 Проверка соответствия теоретическим оценкам

Коэффициент роста t(2n)/t(n) для достаточно больших n:

| Алгоритм / данные | n=4096→8192 | n=8192→16384 | n=16384→32768 | Теория |
|---|---|---|---|---|
| Ins, случайные | 4.02 | 4.07 | 4.06 | **4× (O(n²))** ✓ |
| Heap, случайные | 2.18 | 2.19 | 2.27 | **~2.2× (O(n log n))** ✓ |
| Ins, возрастание | 1.95 | 2.14 | 2.08 | **2× (O(n))** ✓ |
| Heap, возрастание | 2.17 | 2.22 | 2.26 | **~2.2× (O(n log n))** ✓ |
| Ins, убывание | 4.02 | 4.03 | 4.20 | **4× (O(n²))** ✓ |
| Heap, убывание | 2.19 | 2.21 | 2.20 | **~2.2× (O(n log n))** ✓ |

Для n ≥ 4096 экспериментальные данные точно соответствуют теоретическим оценкам сложности.

### 4.2 Шум при малых n

При n < 1024–2048 коэффициенты роста заметно отклоняются от теоретических. Причины:
- **Insertion sort sorted**: время сортировки (~1–2 мкс) сопоставимо с накладными расходами malloc+memcpy (~5–7 мкс), коэффициент роста оказывается < 2.
- **Heap sort**: сложная структура siftDown даёт непостоянный коэффициент при малых n из-за эффектов предсказания ветвлений.
- **Выброс n=512 (insertion, случайные)**: коэффициент 8× вместо 4× — вероятно, граница уровня кэша или паразитный эффект планировщика ОС; воспроизводится не всегда.

### 4.3 Точка пересечения

На **случайных данных**:
- При n=256: Insertion (92 мкс) < Heap (110 мкс) — сортировка вставками быстрее.
- При n=512: Insertion (741 мкс) >> Heap (198 мкс) — пирамидальная уже в 3.7× быстрее.
- Реальная точка пересечения: **n ≈ 256–350** (оценка по теоретическим кривым ~n=115, но константа heap sort при малых n выше из-за сложности siftDown).

На **обратно упорядоченных** данных Heap Sort быстрее уже при **любом** n (даже n=256 в 4.9×).

На **упорядоченных** данных Insertion Sort быстрее при **любом** n (даже n=32768 в 31×).

## 5. Выводы

### Когда эффективнее сортировка вставками

| Условие | Преимущество |
|---|---|
| Данные **уже отсортированы** или **почти** отсортированы | O(n) вместо O(n log n) — в 10–30× быстрее heap sort |
| **Малые массивы** (n ≲ 300) на случайных данных | Меньше накладных расходов, простая структура |
| **Online-сортировка** (элементы поступают по одному) | Можно вставлять «на ходу» — heap sort так не умеет |
| Встроенные системы с минимальной памятью | Простой код, предсказуемые обращения к памяти |

### Когда эффективнее пирамидальная сортировка

| Условие | Преимущество |
|---|---|
| Большие **случайные** массивы (n ≳ 500) | O(n log n) против O(n²) — при n=32768 в 129× быстрее |
| **Обратно отсортированные** данные | Insertion sort деградирует до O(n²), heap не зависит от порядка |
| Нужна **гарантия** O(n log n) в худшем случае | Heap sort не деградирует никогда |
| **Потоковый выбор k минимальных/максимальных** | Структура кучи позволяет эффективно поддерживать top-k |

### Итог

```
Входные данные             | Лучший алгоритм
---------------------------|----------------
Уже отсортированы          | Insertion Sort  (O(n))
Почти отсортированы        | Insertion Sort  (O(n·k), k — количество инверсий)
Случайные, n < ~300        | Insertion Sort  (меньше накладных расходов)
Случайные, n > ~500        | Heap Sort       (O(n log n) vs O(n²))
Обратно отсортированные    | Heap Sort       (всегда, даже при малых n)
Неизвестный порядок        | Heap Sort       (стабильная сложность)
```

> **Практическая рекомендация**: гибридный подход (как в `std::sort` / `Introsort`) — используй сортировку вставками для подмассивов длиной ≤ 16 элементов и переключайся на O(n log n)-алгоритм для больших подмассивов.
